Performance Budget

The performance budget for this site was established to be:

  • The homepage must weigh less than 1mb, and have a speed index under 3000 on a 3G connection.
  • All other pages must weigh less than 650kb, and have a speed index under 1500 on a 3G connection.

This is obviously only an example. A performance budget is a necessary constraint because we do care about user experience and it enforces discipline on design and development.

Using a performance budget is not an excuse to use lowest common denominator design. Just the opposite - how to achieve complex, interesting, effective designs with the least CSS? That's craft.

Be sure to check contrast ratios against WCAG Standards, and test designs in greyscale first! WCAG 2.0 level AA requires a contrast ratio of 4.5:1 for normal text and 3:1 for large text. Level AAA requires a contrast ratio of 7:1 for normal text and 4.5:1 for large text. Large text is defined as 14 point (typically 18.66px) and bold or larger, or 18 point (typically 24px) or larger.

The only colors on here that I didn't check for WCAG contrast are the "text" in the example divs. All else pass AA at least. I simply went to a color palette generator and plugged in that color of red as primary. Color palettes should have a logic behind them, but so long as the basic colors are defined, and tested for contrast, nearly anything goes!

For this example, headings are set in Roboto Condensed at regular, bold, and bold italic. all other body elements are set in Lora regular or regular italic. Both Roboto Condensed and Lora are served from Google Fonts.

When setting Heading type, our chief concern is to establish hierarchy. When setting body type, our chief concern is readability. The best practice is to set body copy first, then set header type to pair. We can complement a serif body with a sans serif header (or vice versa), we can contrast by using two of the same style (serif+serif, sans+sans), or we can use the same for body and heading, if appropriate. The only real guide here is what works for the project, which is why getting real content as soon as possible is so helpful!

How to develop Ryhthm

Very. Carefully.

line-height + margin = multiple

Seriously though, vertical rhythm is difficult to achieve because of different text sizes and layouts. One simple way to achieve basic rhythm is this equation: line-height + margin = multiple. Line height is always relative to font size. Since we created our font size above, with modular scale, we can start there. In this example, I settled on a line height of 1.375 for body copy. This is on the tight end of line-height, but gives us simple math to work with. 16px x 1.375 = 22px. Then, I simply added this amount as margin-bottom to pargraphs, yielding 44px. For headings and other text elements, I simply ensured that the total of the element's line height and margin was a multiple of 44px. This way, every line ends up neatly on the base line. You can certainly also set responsive baselines if you're so inclined, however, I tend to think this is overkill and urge you to remember that no one will see a baseline unless you show them it, and it's a guide more than anything else.

This is a paragraph. Note how size of the letters, length of the line, and line height. There is no one ideal line length on the web, but aiming between 75-100 characters most of the time is a good goal to shoot for. This helps the eye relax and refocus on to the next line. With the text set as above, the ideal container size for paragraph copy is medium-10 large-6 large-offset-3 columns. You will also note that for above the large breakpoints, the aside and the main copy come together into the middle 11 columns. This eases the reading experience and keeps related content together.

Variations

Sometimes, we need to draw special attention to elements via the use of display or special text. Colloquially these are called "Snowflakes", and much like we should not rely on screen size to pre-determine breakpoints, we ought not rely on all heading styles to account for all requirements. If the design requires it, we ought to make variations. Some best practices are to:

  • Define when variations are to be used, and when they are inappropiate
  • Use from the same font service as the other fonts
  • Only load the weight and style the design needs
  • If possible, consider using an svg instead!
  • The preferred naming convention is a css class named after the font you're inserting.
  • The Filament Group, Agency of the Year 2015 and the industry leaders in performant responsive design, suggest limiting font variations to 5 per page load. We've already hit that limit, but this is just an example.

E.g. .suez will change text to the Suez One. Variations ought to be used sparingly and after the existing system is exhausted.

Methods to establish Hierarchy

Here are some of the ways we can show hierarchy with headers:

  • Scale - bigger = more important
  • Weight - heavier = more important
  • Variety - Exception = more important

Factors for Readability

Here are some of the criteria for selecting body type:

  • X-height - taller x height = easier to distinguish from farther away
  • "I, L, J, T" - more distinct = easier to read
  • Kerning - more space between letters = easier to read

Rules of Thumb

Worth remembering (so we can consciously break them)

  • Font-Size 12-21 px.
  • Line-Height Generally between 1-1.4 for headings, 1.25-2 for body.
  • Line-Length Ideal for web is between 75-100 characters
Generally, text set within these bounds will conform to usability guidelines. Consciously using much larger, smaller, shorter, tighter or wider set text will force the user to react to that decision.

Performance

A design will be most reliable if the fonts:

  • Are chosen with care, generally from the same provider
  • Variations are kept to a minimum, ideally, 5 or less per page.
  • Using Font Face Observer to
    1. Load the page first with system fonts
    2. Check to see if the web fonts have loaded in a short window
    3. If fonts have loaded, append .font-loaded to the body, and replace the system fonts with the web fonts

Grid

The grid width is set to: 1600px.

The gutters are set to: 18px: small, 36px: medium up.

The number of columns is set to: 16.

The column width is therefore: 82px small and 64px medium-up.

I didn't put a ton of thought into the dimensions of this grid, other than to demonstrate that even slightly tweaking the dimensions (1200 wide, 12 columns, 15px gutter or 30px medium up) can help ward off the stale feeling of using a framework.

It's important to distinguish here that grid and layout are not the same thing, nor is simply setting the grid a substitute for defining a layout, or, more appropriately for the web, how a layout ought to shift as the web is not fixed to any set of dimensions.

Display Patterns

This is where we'd define styles for buttons, forms, and other UI components, like cards, sliders, widgets, etc. Because user experience depends upon consistency of interaction, standards for variables like border radius and amount of padding, transition times, etc., can and should be decided and recorded here. Again, proactively deciding this now saves time in development and bloat in production!

The exercise of doing this stuff manually helps me to understand what I'm building at a deeper level.

There are style guide generators available, like Pattern Lab or Spry's own Nimble Builder, but I'm old school and the exercise of doing this stuff manually helps me to understand what I'm building at a deeper level.The better a front end developer understands the design intent, the easier it is to not just produce the site, but to produce it in a performant, semantic, accessible, progressively enhanced manner. In short, the closer the front-end developer and the designer work, the better the site is for everyone.

Buttons

Only use buttons to indicate something a user should touch to initiate or complete an action. Buttons are not anchors and not inputs on forms, but in very rare cases, you can apply the css class to achieve the desired effect. Only other note is to make it easy on your developer and pick a global border-radius, font-size, style, and padding.
